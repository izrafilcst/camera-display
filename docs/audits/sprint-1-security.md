## 1. Overall posture

**NEEDS HARDENING** — Sprint 1 is a display bring-up with **no network surface yet**, so the residual risk is low for this slice. There are **no Critical or High findings**. However, the project is shipping with **all ESP-IDF security defaults left implicit** (secure-boot, flash encryption, brownout, stack-canary protections not explicitly enabled in `sdkconfig.defaults`), and several hardening primitives that *must* be in place before Sprint 2 (ESP-NOW parser) are still absent. Sign-off is conditional on landing the Medium items below before any radio code merges.

## 4. Sprint 2 attack-surface preview

Sprint 2 introduces ESP-NOW reception and JPEG decode. The following will become the dominant attack surface — track these *now* so the parser code lands hardened on day one:

1. **`tx_emission_ms u32` + variable-length fragment header** (spec §4.2, line 124-125). Header is `frame_id u16 / frag_idx u8 / frag_total u8 / jpeg_size u16 / payload_len u16` + conditional 4 B of `tx_emission_ms` when `frag_idx == 0`. **Risks**: (a) `payload_len` vs actual frame size mismatch → OOB read, (b) `frag_total == 0` → division/modulo by zero in reassembly, (c) `frag_idx >= frag_total` → OOB write into slot, (d) `jpeg_size` larger than 16 KB slot → heap overflow.  *Mitigation to require:* explicit bound check on every field at entry, including `frag_idx < MAX_FRAGS_PER_FRAME` (compile-time const).

2. **JPEG decoder fuzzing surface**. `TJpgDec` and `esp_jpeg` have a history of crash bugs on malformed input. Even without network attacker, garbage from interference can crash the decode task. *Mitigation to require:* run decode in its own task with a shallow stack-watermark, plus `esp_task_wdt_add()` so a hung decoder reboots cleanly.

3. **Reassembly slot replay / amplification**. Spec §4.2 keeps a slot alive 30 ms then drops. An attacker spraying many `frame_id` values can exhaust both slots (`2 × 16 KB`) and freeze pipeline. *Mitigation to require:* rate-limit `new slot` creation per source MAC; drop oldest, not newest.

4. **Hardcoded peer MAC (V0)**. Spec §6, line 261 makes this explicit. Anyone within RF range can spoof packets. *Mitigation:* file an issue now to (a) gate at compile time behind `CONFIG_RECEIVER_PEER_MAC` Kconfig, (b) emit a boot-time `ESP_LOGW` if the MAC is the placeholder, (c) add a unit test that fails CI if the MAC matches a list of known placeholders.

5. **Telemetry back-channel** (spec §6, line 268, `uint8_t seq` "ordering anti-replay"). A single-byte seq wraps every 256 packets — trivially replayable. *Mitigation:* widen to `uint32_t` *now* in the spec; record last-seen seq per peer; reject decreasing seq (modulo a small reorder window).

## 5. Overall security posture

**NEEDS HARDENING** — gated, not blocking.

Sprint 1 itself has **zero Critical/High findings**, no hardcoded credentials, no untrusted build inputs, and the only network-adjacent code (LovyanGFX) is a well-known library accessed locally over SPI. The host-test suite uses `-Wall -Wextra -Werror`, which is excellent practice and should be propagated.

The audit is conditioned on Sprint 2 not landing until at least items **S-01** (brownout pin), **S-02** (stack canaries), and the four header-validation requirements in §4 above are merged. Items **S-03**/**S-04** (secure boot, OTA) are explicitly deferred to Sprint 7 per the existing plan, and we concur with that schedule.

**Recommendation: APPROVE Sprint 1 merge to `main`. Block Sprint 2 merge until S-01/S-02 land and a fragment-parser fuzz harness is in place.**
